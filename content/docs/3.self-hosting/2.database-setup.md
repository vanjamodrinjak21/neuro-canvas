# Database Setup

NeuroCanvas uses **PostgreSQL** as its primary database and **Redis** as an optional caching layer. The database schema is managed by [Prisma ORM](https://www.prisma.io/).

## PostgreSQL Requirements

- **PostgreSQL 15+** (16 recommended)
- The `pgcrypto` extension is not required — ID generation uses CUIDs at the application level
- UTF-8 encoding (default)

## Creating the Database

### Option 1: Docker (Recommended)

The `docker-compose.yml` included in the repository handles database creation automatically:

```bash
docker compose up -d db
```

This creates a PostgreSQL 16 instance with:
- **User:** `postgres`
- **Password:** `postgres` (change this in production)
- **Database:** `neurocanvas`
- **Port:** 5432
- **Data volume:** `postgres_data` (persisted across container restarts)

### Option 2: Manual Setup

If you are using an existing PostgreSQL server:

```sql
-- Connect as a superuser or a user with CREATEDB privileges
CREATE DATABASE neurocanvas;

-- Create a dedicated user (recommended)
CREATE USER neurocanvas_user WITH ENCRYPTED PASSWORD 'your-strong-password';
GRANT ALL PRIVILEGES ON DATABASE neurocanvas TO neurocanvas_user;

-- Connect to the new database and grant schema permissions
\c neurocanvas
GRANT ALL ON SCHEMA public TO neurocanvas_user;
ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT ALL ON TABLES TO neurocanvas_user;
ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT ALL ON SEQUENCES TO neurocanvas_user;
```

Set `DATABASE_URL` in your environment:
```
DATABASE_URL="postgresql://neurocanvas_user:your-strong-password@localhost:5432/neurocanvas"
```

### Option 3: Managed PostgreSQL

NeuroCanvas works with managed PostgreSQL providers:

- **Neon** — serverless PostgreSQL with branching
- **Supabase** — PostgreSQL with built-in auth (use only the database, not Supabase Auth)
- **Railway** — one-click PostgreSQL provisioning
- **AWS RDS** / **Google Cloud SQL** / **Azure Database for PostgreSQL**

Use the connection string provided by your hosting provider as the `DATABASE_URL`. Ensure SSL is enabled for remote connections:

```
DATABASE_URL="postgresql://user:password@host:5432/neurocanvas?sslmode=require"
```

## Running Migrations

Prisma manages the database schema. On first setup or after updates, apply the schema:

### Docker Deployments

The Docker entrypoint script runs migrations automatically on container start:

```bash
# Runs automatically inside the container:
npx prisma db push --skip-generate --accept-data-loss
```

No manual intervention is needed for Docker deployments.

### Manual Deployments

```bash
# Apply migrations to the database
npx prisma migrate deploy

# Alternative: push schema directly (for development or initial setup)
npx prisma db push
```

::callout{type="info"}
`prisma migrate deploy` applies migration files from the `prisma/migrations/` directory and is the recommended approach for production. `prisma db push` synchronizes the schema directly without creating migration files and is better suited for development.
::

### Generating the Prisma Client

After schema changes, regenerate the TypeScript client:

```bash
npx prisma generate
```

This is handled automatically during `npm install` (via the `postinstall` script) and during Docker builds.

## Schema Overview

The database schema consists of the following tables:

### Authentication Tables

These tables are required by NextAuth.js (Auth.js) for session management and OAuth:

| Table | Purpose |
|-------|---------|
| `User` | User accounts. Contains `id` (CUID), `name`, `email`, `bio`, `emailVerified`, `image`, hashed `password` (for credentials auth), `passwordChangedAt`, encrypted `totpSecret`, `totpEnabled` flag, hashed `backupCodes` array, and timestamps. |
| `Account` | Linked OAuth provider accounts. One user can have multiple accounts (e.g., Google + GitHub). Cascades on user deletion. |
| `Session` | Active sessions with token and expiry. Cascades on user deletion. |
| `VerificationToken` | Email verification and password reset tokens. Unique on `[identifier, token]`. |

### Map Storage Tables

| Table | Purpose |
|-------|---------|
| `Map` | Mind maps. Uses client-generated CUID as primary key. Stores the full map data as JSON (`nodes`, `edges`, `camera`, `settings`, `rootNodeId`). Includes a SHA-256 `checksum` for integrity verification and `syncVersion` for optimistic concurrency control. Supports soft-delete via `deletedAt`. Indexed on `[userId, updatedAt]`. |
| `MapVersion` | Version history for maps. Each save creates a new version entry. Unique on `[mapId, version]`. Cascades on map deletion. |
| `SyncState` | Per-user sync metadata. Tracks `lastSyncAt`, `lastDeviceId`, `totalMaps`, and `totalByteSize`. One-to-one with `User`. |

### Sharing Tables

| Table | Purpose |
|-------|---------|
| `MapShare` | Shareable link tokens for maps. Contains `token` (unique), `mapId`, `userId`, optional `expiresAt`, and timestamps. Allows public read-only access without authentication. |

### Template Tables

| Table | Purpose |
|-------|---------|
| `Template` | Reusable map structures. Contains `slug` (unique), `title`, `description`, `category` (education/business/creative/planning/research), `tags`, `nodeCount`, `levelCount`, `nodes`/`edges`/`settings` (JSON), `previewData` (JSON), `aiEnhanced` flag, `isSystem`/`isPublic` flags, `usageCount`, and optional `authorId`. Indexed on `[category]`. |

### Credential Vault

| Table | Purpose |
|-------|---------|
| `Credential` | Encrypted API keys for AI providers (OpenAI, Anthropic, OpenRouter, custom). Keys are encrypted server-side and never returned in plaintext. Unique on `[userId, provider, label]`. |

### Entity Relationship Diagram

```
User
 ├── Account[]         (1:N, CASCADE)
 ├── Session[]         (1:N, CASCADE)
 ├── Credential[]      (1:N, CASCADE)
 ├── Map[]             (1:N, CASCADE)
 ├── Template[]        (1:N, SET NULL on authorId)
 └── SyncState?        (1:1, CASCADE)

Map
 ├── MapVersion[]      (1:N, CASCADE)
 └── MapShare[]        (1:N, CASCADE)
```

All relationships use `ON DELETE CASCADE` unless noted — deleting a user removes all their data. Template `authorId` is set to null on user deletion so system templates persist.

## Backup and Restore

### Automated Backups with pg_dump

```bash
# Backup
pg_dump -h localhost -U postgres -d neurocanvas -F c -f backup_$(date +%Y%m%d_%H%M%S).dump

# Restore
pg_restore -h localhost -U postgres -d neurocanvas -c backup_20260324_120000.dump
```

### Docker Backup

```bash
# Backup from the running container
docker compose exec db pg_dump -U postgres neurocanvas > backup.sql

# Restore
docker compose exec -T db psql -U postgres neurocanvas < backup.sql
```

### Scheduled Backups

Use a cron job for automated daily backups:

```bash
# Add to crontab (crontab -e)
0 3 * * * pg_dump -h localhost -U postgres -d neurocanvas -F c -f /backups/neurocanvas_$(date +\%Y\%m\%d).dump 2>&1 | logger -t neurocanvas-backup
```

Retain at least 7 daily backups and test restoration periodically.

## Connection Pooling

For production deployments with multiple application instances or serverless environments, use a connection pooler.

### PgBouncer

PgBouncer sits between your application and PostgreSQL, pooling and reusing database connections:

```bash
# Install PgBouncer
sudo apt install pgbouncer
```

Example `pgbouncer.ini`:
```ini
[databases]
neurocanvas = host=localhost port=5432 dbname=neurocanvas

[pgbouncer]
listen_addr = 127.0.0.1
listen_port = 6432
auth_type = md5
auth_file = /etc/pgbouncer/userlist.txt
pool_mode = transaction
max_client_conn = 200
default_pool_size = 20
```

Update `DATABASE_URL` to point to PgBouncer:
```
DATABASE_URL="postgresql://postgres:postgres@localhost:6432/neurocanvas"
```

::callout{type="info"}
Use `pool_mode = transaction` (not `session`) for Prisma compatibility. Prisma uses named prepared statements which are scoped to sessions, but transaction mode handles this correctly with Prisma's `pgbouncer=true` flag.
::

For Prisma with PgBouncer, add the `pgbouncer` parameter to the connection string:
```
DATABASE_URL="postgresql://postgres:postgres@localhost:6432/neurocanvas?pgbouncer=true"
```

### Prisma Connection Limits

Prisma uses a connection pool internally. Configure pool size via the connection string:

```
DATABASE_URL="postgresql://postgres:postgres@localhost:5432/neurocanvas?connection_limit=10"
```

The default is 10 connections. For single-instance deployments, 10-20 is typically sufficient. Reduce this if using PgBouncer.

## Redis Setup

Redis is used for caching and is optional but recommended for production.

### What Redis Caches

Redis is used for:

- **Embedding vectors** for semantic similarity computation
- **AI response cache** to avoid redundant LLM API calls
- **Sync state** for fast reads without hitting PostgreSQL
- **Distributed locks** for concurrent map edit safety
- **Rate limiting** counters for API abuse prevention
- **Sync event streams** for real-time change notifications
- **Metrics** for monitoring sync operations

### Docker Redis

The included `docker-compose.yml` runs Redis 7 Alpine with append-only file persistence:

```yaml
redis:
  image: redis:7-alpine
  command: redis-server --appendonly yes
  volumes:
    - redis_data:/data
```

This provides durability across container restarts. For ephemeral caching where data loss is acceptable, remove the volume mount.

### Managed Redis

NeuroCanvas works with managed Redis services:

- **Upstash** — serverless Redis with per-request pricing
- **Redis Cloud** — managed Redis from Redis Ltd.
- **AWS ElastiCache** / **Google Memorystore** / **Azure Cache for Redis**

Set the `REDIS_URL` environment variable to the connection string provided by your hosting provider. For TLS connections, use the `rediss://` protocol.

### Running Without Redis

If `REDIS_URL` is not set or Redis is unreachable, the application falls back gracefully:

- Cache reads return `null` (cache miss)
- Cache writes are silently dropped
- The application continues to function, but with higher latency on repeated operations
- Sync rate limiting and distributed locks are disabled
