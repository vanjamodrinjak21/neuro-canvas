# Authentication

NeuroCanvas uses [NextAuth.js](https://next-auth.js.org/) (integrated via `@sidebase/nuxt-auth`) for authentication. The system supports multiple providers and uses JWT-based sessions backed by a PostgreSQL database through the Prisma adapter.

## Auth Flow Overview

```
User clicks "Sign In"
    |
    v
Provider selection (Google, GitHub, Email, or Credentials)
    |
    v
OAuth redirect / Credentials validation / Magic link
    |
    v
NextAuth callback -> JWT created -> Session cookie set
    |
    v
User redirected to /dashboard (new users) or previous page
```

### Session Strategy

NeuroCanvas uses JWT-based sessions stored in secure HTTP-only cookies. Sessions expire automatically and are refreshed during active use.

### Database Adapter

The database adapter persists users, accounts, and verification tokens. This enables:

- Linking multiple OAuth providers to a single user account
- Secure password storage
- Email verification token management

## Available Providers

### Google OAuth

Sign in with Google using the OAuth 2.0 authorization code flow.

- Requires `GOOGLE_CLIENT_ID` and `GOOGLE_CLIENT_SECRET` environment variables
- Requests offline access with `access_type: "offline"` for refresh tokens
- Email account linking is enabled (`allowDangerousEmailAccountLinking: true`)

### GitHub OAuth

Sign in with GitHub using OAuth.

- Requires `GITHUB_CLIENT_ID` and `GITHUB_CLIENT_SECRET` environment variables
- Requests `read:user` and `user:email` scopes
- Automatically fetches primary email from the GitHub emails API if not included in the profile
- Email account linking is enabled

### Email (Magic Link)

Passwordless sign-in via email using [Resend](https://resend.com/) for transactional email delivery.

- Requires `RESEND_API_KEY` and email sender configuration
- Sends a branded HTML email with a one-click sign-in link
- Links expire after 24 hours
- Only available when Resend is configured

### Credentials (Email + Password)

Traditional email and password authentication.

- Always available regardless of environment configuration
- Passwords are securely hashed before storage
- Passwords must meet minimum strength requirements (length, character variety)
- Passwords are checked against known data breach databases on registration and password change
- If two-factor authentication is enabled, a valid TOTP code is required at sign-in

## API Endpoints

### Get Current Session

Retrieve the currently authenticated user's session.

```
GET /api/auth/session
```

**Response (authenticated):**
```json
{
  "user": {
    "id": "clx1abc2d0001",
    "email": "user@example.com",
    "name": "Jane Doe",
    "image": "https://lh3.googleusercontent.com/..."
  },
  "expires": "2026-01-15T14:30:00.000Z"
}
```

**Response (unauthenticated):**
```json
{}
```

### List Available Providers

Returns the authentication providers configured on the server.

```
GET /api/auth/providers
```

**Response:**
```json
{
  "google": {
    "id": "google",
    "name": "Google",
    "type": "oauth",
    "signinUrl": "/api/auth/signin/google",
    "callbackUrl": "/api/auth/callback/google"
  },
  "github": {
    "id": "github",
    "name": "GitHub",
    "type": "oauth",
    "signinUrl": "/api/auth/signin/github",
    "callbackUrl": "/api/auth/callback/github"
  },
  "credentials": {
    "id": "credentials",
    "name": "Credentials",
    "type": "credentials",
    "signinUrl": "/api/auth/signin/credentials",
    "callbackUrl": "/api/auth/callback/credentials"
  }
}
```

The response only includes providers whose environment variables are configured. For example, Google will not appear if `GOOGLE_CLIENT_ID` is missing.

### Sign In

Initiate a sign-in flow. The method depends on the provider type.

#### OAuth Providers (Google, GitHub)

```
POST /api/auth/signin/:provider
```

Redirects the user to the OAuth provider's authorization page. After consent, the user is redirected back to `/api/auth/callback/:provider`, which completes the JWT exchange and sets the session cookie.

#### Credentials Provider

```
POST /api/auth/signin/credentials
Content-Type: application/json

{
  "email": "user@example.com",
  "password": "SecureP4ss"
}
```

**Success:** Sets session cookie and redirects to `/dashboard`.

**Failure (401):**
```json
{
  "error": "CredentialsSignin",
  "url": "/auth/error?error=CredentialsSignin"
}
```

### Register (Credentials)

Create a new account with email and password. This is a custom endpoint outside of the NextAuth.js flow.

```
POST /api/auth/register
Content-Type: application/json

{
  "email": "user@example.com",
  "password": "SecureP4ss",
  "name": "Jane Doe"
}
```

**Success (200):**
```json
{
  "success": true,
  "user": {
    "id": "clx1abc2d0001",
    "email": "user@example.com",
    "name": "Jane Doe",
    "createdAt": "2026-01-15T10:30:00.000Z"
  }
}
```

**Errors:**

| Status | Condition |
|---|---|
| `400` | Missing email or password |
| `400` | Invalid email format |
| `400` | Password does not meet requirements |
| `400` | Password found in known data breaches |
| `409` | Account with this email already exists |
| `429` | Rate limit exceeded |

### Sign Out

Terminate the current session.

```
POST /api/auth/signout
```

Clears the session cookie and redirects to `/auth/signin`.

### CSRF Token

Retrieve a CSRF token for use in form submissions.

```
GET /api/auth/csrf
```

**Response:**
```json
{
  "csrfToken": "abc123..."
}
```

## Session Usage in API Requests

All protected server routes validate the session using `getServerSession(event)` from the `#auth` module:

```typescript
import { getServerSession } from '#auth'

export default defineEventHandler(async (event) => {
  const session = await getServerSession(event)

  if (!session?.user?.email) {
    throw createError({
      statusCode: 401,
      statusMessage: 'Unauthorized'
    })
  }

  // Access session.user.id, session.user.email, etc.
})
```

The session cookie is automatically included in browser requests. No manual `Authorization` header is needed for web app usage.

## User Profile Endpoints

### Get Profile

```
GET /api/user/profile
```

Returns the authenticated user's profile along with their linked OAuth providers.

**Response:**
```json
{
  "id": "clx1abc2d0001",
  "email": "user@example.com",
  "name": "Jane Doe",
  "bio": "Researcher and mind mapping enthusiast",
  "image": "https://lh3.googleusercontent.com/...",
  "emailVerified": "2026-01-01T00:00:00.000Z",
  "totpEnabled": false,
  "createdAt": "2026-01-01T00:00:00.000Z",
  "updatedAt": "2026-01-15T10:30:00.000Z",
  "providers": ["google", "github"]
}
```

### Update Profile

```
PATCH /api/user/profile
Content-Type: application/json
X-Requested-With: XMLHttpRequest

{
  "name": "Jane Smith",
  "bio": "Researcher and mind mapping enthusiast",
  "image": "https://example.com/avatar.jpg"
}
```

All fields are optional. Name is capped at 100 characters. Bio is capped at 280 characters. Image must be an HTTPS URL.

**Response:**
```json
{
  "id": "clx1abc2d0001",
  "email": "user@example.com",
  "name": "Jane Smith",
  "bio": "Researcher and mind mapping enthusiast",
  "image": "https://example.com/avatar.jpg",
  "updatedAt": "2026-01-15T11:00:00.000Z"
}
```

### Change Password

Change the password for a credentials-based account. Requires the current password for verification. The new password is checked against HaveIBeenPwned. Changing your password invalidates all existing sessions.

```
POST /api/user/change-password
Content-Type: application/json
X-Requested-With: XMLHttpRequest

{
  "currentPassword": "OldP4ssword",
  "newPassword": "NewSecureP4ss"
}
```

**Success (200):**
```json
{
  "success": true,
  "message": "Password changed successfully"
}
```

**Errors:**

| Status | Condition |
|---|---|
| `400` | Missing current or new password |
| `400` | New password does not meet requirements |
| `400` | New password found in known data breaches |
| `401` | Current password is incorrect |
| `403` | Account does not use password authentication |
| `429` | Rate limit exceeded |

### Export Account Data

Download all user data as a JSON file.

```
GET /api/user/export
```

**Response:** A JSON file download containing profile data, all maps (with nodes, edges, and metadata), published templates, and credential metadata (provider names only -- no encrypted keys).

This endpoint is rate-limited.

### Two-Factor Authentication (TOTP)

#### Set Up 2FA

Start the TOTP setup process. Returns a QR code and secret key for authenticator app registration.

```
POST /api/user/totp/setup
X-Requested-With: XMLHttpRequest
```

**Response:**
```json
{
  "qrCode": "data:image/png;base64,...",
  "uri": "otpauth://totp/NeuroCanvas:user@example.com?secret=..."
}
```

#### Verify and Enable 2FA

Verify a TOTP code from the authenticator app to activate 2FA. Returns one-time backup codes.

```
POST /api/user/totp/verify
Content-Type: application/json
X-Requested-With: XMLHttpRequest

{
  "code": "123456"
}
```

**Response:**
```json
{
  "success": true,
  "backupCodes": ["...", "..."]
}
```

::callout{type="warning"}
Save your backup codes in a safe place. Each code can only be used once. If you lose access to your authenticator app and your backup codes, you will be locked out of your account.
::

#### Disable 2FA

Disable two-factor authentication. Requires password re-authentication.

```
POST /api/user/totp/disable
Content-Type: application/json
X-Requested-With: XMLHttpRequest

{
  "password": "YourP4ssword"
}
```

**Response:**
```json
{
  "success": true,
  "message": "Two-factor authentication disabled"
}
```

### Delete Account

Permanently delete the authenticated user's account and all associated data (maps, sessions, OAuth accounts). This action cascades through all related Prisma models.

```
DELETE /api/user/account
Content-Type: application/json

{
  "confirmation": "DELETE MY ACCOUNT"
}
```

The exact string `"DELETE MY ACCOUNT"` must be provided as confirmation. Any other value returns a `400` error.

**Response:**
```json
{
  "success": true,
  "message": "Account deleted successfully"
}
```

## Security Notes

- All session cookies use `HttpOnly`, `SameSite`, and `Secure` flags in production
- The `AUTH_SECRET` environment variable is required and must be a strong random string
- OAuth tokens are stored encrypted in the database
- Passwords are securely hashed before storage
- Passwords are checked against known breach databases on registration and password change
- Changing a password invalidates existing sessions
- Two-factor authentication uses standard TOTP with one-time backup codes
- CSRF protection is enforced on all state-changing endpoints
- API keys in the credential vault are encrypted server-side
