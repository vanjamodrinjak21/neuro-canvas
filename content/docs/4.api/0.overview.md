# API Overview

The NeuroCanvas API is a RESTful JSON API built on Nuxt 3 server routes. All endpoints live under `/api` and follow consistent conventions for authentication, error handling, and response structure.

## Base URL

| Environment | Base URL |
|---|---|
| Production | `https://neuro-canvas.com/api` |
| Local development | `http://localhost:3000/api` |
| Self-hosted | `https://your-domain.com/api` |

All requests and responses use `Content-Type: application/json` unless noted otherwise (streaming endpoints use `text/event-stream`).

## Authentication

Most endpoints require an authenticated session. NeuroCanvas uses [NextAuth.js](https://next-auth.js.org/) (via `@sidebase/nuxt-auth`) with a JWT session strategy.

### Obtaining a session token

After signing in through one of the available providers (Google, GitHub, Email, or Credentials), the server sets a secure HTTP-only session cookie. Browser requests automatically include this cookie. No manual `Authorization` header is needed for web app usage.

### Session validation

Every protected endpoint validates the session server-side. If the session is missing or expired, the server returns a `401 Unauthorized` response. Sessions have a configurable expiry and are automatically refreshed during active use.

## Rate Limiting

All sensitive API endpoints are rate-limited to prevent abuse. Rate limits are enforced per-user or per-IP depending on the endpoint. The following operations are rate-limited:

- Sync operations
- Registration
- Password changes
- Account deletion
- Data export
- TOTP setup and verification
- AI endpoints

When a rate limit is exceeded, the server returns `429 Too Many Requests`.

## Response Format

### Successful responses

NeuroCanvas API responses are flat JSON objects. The shape depends on the endpoint, but common patterns include:

**Single resource:**
```json
{
  "id": "clx1abc2d0001",
  "email": "user@example.com",
  "name": "Jane Doe",
  "createdAt": "2026-01-15T10:30:00.000Z"
}
```

**Collection with metadata:**
```json
{
  "maps": [
    { "id": "map_1", "title": "Project Roadmap", "syncVersion": 3 }
  ],
  "deleted": ["map_old_1"]
}
```

**Operation result:**
```json
{
  "success": true,
  "stored": 12
}
```

### Paginated responses

Sync pull endpoints support time-based pagination using the `since` query parameter (ISO 8601 timestamp). The response includes all records updated after the specified time:

```
GET /api/sync/pull?since=2026-01-01T00:00:00.000Z
```

### Error responses

All errors follow the Nuxt/H3 error format with an HTTP status code and a human-readable message:

```json
{
  "statusCode": 400,
  "statusMessage": "Missing required fields: mapId, data, title, checksum"
}
```

AI proxy endpoints return errors with a `message` field instead:

```json
{
  "statusCode": 500,
  "message": "AI request failed"
}
```

## HTTP Status Codes

| Code | Meaning | When used |
|---|---|---|
| `200` | OK | Successful GET, PATCH, POST, or DELETE |
| `201` | Created | Not currently used (successful creation returns 200) |
| `400` | Bad Request | Missing or invalid parameters, validation failure |
| `401` | Unauthorized | Missing or expired session |
| `403` | Forbidden | Valid session but insufficient permissions |
| `404` | Not Found | Resource does not exist |
| `409` | Conflict | Sync version conflict (server has a newer version) or duplicate resource |
| `423` | Locked | Map is locked by another sync operation |
| `429` | Too Many Requests | Rate limit exceeded |
| `500` | Internal Server Error | Unexpected server failure |

## CSRF Protection

NeuroCanvas applies multiple layers of CSRF protection on all state-changing endpoints. This includes cookie-based CSRF tokens and custom header validation. Cross-origin form submissions are blocked.

## Health Check

A public health check endpoint is available without authentication:

```
GET /api/health
```

```json
{
  "status": "ok",
  "timestamp": "2026-01-15T10:30:00.000Z",
  "version": "1.0.0"
}
```

## Caching

NeuroCanvas uses server-side caching for map data, AI responses, embeddings, and sync state. Cache is automatically invalidated on write operations (push, update, delete).
