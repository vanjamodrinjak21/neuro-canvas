# AI Endpoints

NeuroCanvas provides AI features through a server-side proxy layer. The proxy forwards requests to the user's configured AI provider (OpenAI, Anthropic, OpenRouter, Ollama, or a custom OpenAI-compatible endpoint), avoiding CORS issues and keeping API keys on the server side.

All AI endpoints require the client to supply the provider name and API key in the request body. Keys are stored client-side in an encrypted IndexedDB vault and sent per-request.

## Supported Providers

| Provider | Default Base URL | Default Model |
|---|---|---|
| `openai` | `https://api.openai.com/v1` | `gpt-4o-mini` |
| `anthropic` | `https://api.anthropic.com/v1` | `claude-sonnet-4-20250514` |
| `openrouter` | `https://openrouter.ai/api/v1` | `openai/gpt-4o-mini` |
| `ollama` | `http://localhost:11434` | `llama3.2` |
| `custom` | User-provided (required) | `default` |

All providers accept an optional `baseUrl` override for self-hosted or alternative endpoints.

## Completions

Generate a non-streaming AI response.

```
POST /api/ai/completions
Content-Type: application/json

{
  "provider": "openai",
  "apiKey": "sk-...",
  "model": "gpt-4o-mini",
  "messages": [
    { "role": "user", "content": "Suggest 3 subtopics for 'Machine Learning'" }
  ],
  "systemPrompt": "You are a mind mapping assistant. Respond with structured JSON.",
  "maxTokens": 500,
  "temperature": 0.7
}
```

**Parameters:**

| Field | Type | Required | Default | Description |
|---|---|---|---|---|
| `provider` | `string` | Yes | | Provider identifier (see table above) |
| `apiKey` | `string` | Yes | | API key for the provider |
| `model` | `string` | No | Provider default | Model identifier |
| `messages` | `array` | Yes | | Chat messages in `[{ role, content }]` format |
| `systemPrompt` | `string` | No | | System message prepended to the conversation |
| `maxTokens` | `number` | No | `500` | Maximum tokens in the response |
| `temperature` | `number` | No | `0.7` | Sampling temperature (0.0 - 2.0) |
| `baseUrl` | `string` | No | Provider default | Override the provider's base URL |

**Response:**
```json
{
  "content": "Here are 3 subtopics for Machine Learning:\n1. Supervised Learning\n2. Neural Networks\n3. Reinforcement Learning",
  "usage": {
    "prompt_tokens": 42,
    "completion_tokens": 38,
    "total_tokens": 80
  }
}
```

The `usage` field is `null` for Ollama responses, which do not report token usage.

**Errors:**

| Status | Condition |
|---|---|
| `400` | Missing `provider` or `apiKey` |
| `400` | Custom provider missing `baseUrl` |
| `400` | Unsupported provider |
| Upstream status | Forwarded from the AI provider (e.g., 401 for invalid key, 429 for rate limit) |
| `500` | Unexpected failure in the proxy |

## Streaming Completions

Generate a streaming AI response delivered as Server-Sent Events (SSE).

```
POST /api/ai/stream
Content-Type: application/json

{
  "provider": "anthropic",
  "apiKey": "sk-ant-...",
  "model": "claude-sonnet-4-20250514",
  "messages": [
    { "role": "user", "content": "Explain quantum computing in simple terms" }
  ],
  "systemPrompt": "You are a helpful mind mapping assistant.",
  "maxTokens": 500,
  "temperature": 0.7
}
```

The request body is identical to the completions endpoint.

**Response headers:**
```
Content-Type: text/event-stream
Cache-Control: no-cache
Connection: keep-alive
```

**SSE event format:**

Each event is a JSON object on a `data:` line. Three event types are emitted:

**Delta (content chunk):**
```
data: {"type":"delta","content":"Quantum computing uses"}

data: {"type":"delta","content":" quantum bits, or qubits,"}
```

**Done (stream complete):**
```
data: {"type":"done","usage":{"prompt_tokens":42,"completion_tokens":120,"total_tokens":162}}
```

**Error:**
```
data: {"type":"error","message":"API error: 429"}
```

The `usage` field in the `done` event is `null` when the provider does not report usage (Ollama) or when the upstream provider sends usage separately.

### Client-Side Consumption

```typescript
const response = await fetch('/api/ai/stream', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ provider, apiKey, model, messages })
})

const reader = response.body.getReader()
const decoder = new TextDecoder()

while (true) {
  const { done, value } = await reader.read()
  if (done) break

  const text = decoder.decode(value, { stream: true })
  const lines = text.split('\n')

  for (const line of lines) {
    if (line.startsWith('data: ')) {
      const event = JSON.parse(line.slice(6))
      if (event.type === 'delta') {
        // Append event.content to the UI
      } else if (event.type === 'done') {
        // Stream complete
      } else if (event.type === 'error') {
        // Handle error
      }
    }
  }
}
```

## AI Response Cache

NeuroCanvas caches AI responses in Redis to avoid redundant API calls for identical prompts.

### Retrieve Cached Response

```
GET /api/ai/cache?hash=abc123
```

**Response (cache hit):**
```json
{
  "hit": true,
  "response": "Cached AI response text...",
  "usage": { "prompt_tokens": 42, "completion_tokens": 38 },
  "cachedAt": 1718444400000
}
```

**Response (cache miss):**
```json
{
  "hit": false,
  "response": null,
  "usage": null
}
```

### Store Response in Cache

```
POST /api/ai/cache
Content-Type: application/json

{
  "hash": "abc123",
  "response": "The AI response text to cache",
  "usage": { "prompt_tokens": 42, "completion_tokens": 38 },
  "ttlSeconds": 3600
}
```

**Parameters:**

| Field | Type | Required | Default | Description |
|---|---|---|---|---|
| `hash` | `string` | Yes | | Deterministic hash of the prompt |
| `response` | `string` | Yes | | The AI response text |
| `usage` | `object` | No | `null` | Token usage metadata |
| `ttlSeconds` | `number` | No | `3600` | Cache TTL in seconds (default 1 hour) |

**Response:**
```json
{
  "success": true
}
```

## AI User Memory

Persist and retrieve user-specific AI preferences and context in Redis.

### Get Memory

```
GET /api/ai/memory?userId=clx1abc2d0001
```

**Response:**
```json
{
  "data": {
    "preferredStyle": "concise",
    "domainContext": "software-engineering",
    "lastTopics": ["machine-learning", "system-design"]
  }
}
```

Returns `{ "data": null }` if no memory is stored.

### Store Memory

```
POST /api/ai/memory
Content-Type: application/json

{
  "userId": "clx1abc2d0001",
  "data": {
    "preferredStyle": "concise",
    "domainContext": "software-engineering",
    "lastTopics": ["machine-learning", "system-design"]
  }
}
```

Memory entries have a 30-day TTL and are automatically cleaned up.

**Response:**
```json
{
  "success": true
}
```

## Connection Test

Test connectivity to an AI provider and optionally retrieve available models.

```
POST /api/ai/test-connection
Content-Type: application/json

{
  "provider": "openai",
  "apiKey": "sk-...",
  "baseUrl": "https://api.openai.com/v1"
}
```

**Response (success):**
```json
{
  "success": true,
  "message": "API key verified"
}
```

**Response (success with models — Ollama, OpenRouter):**
```json
{
  "success": true,
  "message": "Connected successfully",
  "models": [
    { "id": "llama3.2", "name": "llama3.2" },
    { "id": "mistral", "name": "mistral" }
  ]
}
```

**Response (failure):**
```json
{
  "success": false,
  "message": "Invalid API key"
}
```

### Provider-Specific Behavior

| Provider | Test Method | Returns Models |
|---|---|---|
| `openai` | `GET /models` with Bearer auth | No |
| `anthropic` | Sends a minimal completion request (1 token) | No |
| `openrouter` | `GET /models` with Bearer auth | Yes (up to 50) |
| `ollama` | `GET /api/tags` | Yes (all local models) |
| `custom` | `HEAD` request to the base URL | No |

For Anthropic, a `429 Too Many Requests` response is treated as a successful connection test (the key is valid, just rate-limited).

## Embeddings

Store and retrieve semantic embeddings for map nodes. Embeddings are generated client-side (via transformers.js or the Tauri ML bridge) and cached server-side in Redis for fast retrieval.

### Store Embeddings

```
POST /api/embeddings
Content-Type: application/json

{
  "mapId": "clx1map001",
  "embeddings": [
    {
      "nodeId": "node_abc",
      "text": "Machine Learning",
      "embedding": [0.012, -0.034, 0.056, ...]
    },
    {
      "nodeId": "node_def",
      "text": "Neural Networks",
      "embedding": [-0.023, 0.045, 0.078, ...]
    }
  ]
}
```

Requires authentication. Each embedding is stored with two Redis keys:
1. **By text hash** — for content-addressable lookup across maps
2. **By node ID within map** — for map-specific retrieval

Embeddings have a 7-day TTL.

**Response:**
```json
{
  "success": true,
  "stored": 2
}
```

Entries missing `nodeId`, `text`, or `embedding` are silently skipped.

### Retrieve Embeddings

```
GET /api/embeddings?mapId=clx1map001&nodeIds=node_abc,node_def,node_ghi
```

**Query parameters:**

| Parameter | Type | Description |
|---|---|---|
| `mapId` | `string` | The map ID to scope the lookup |
| `nodeIds` | `string` | Comma-separated list of node IDs |

**Response:**
```json
{
  "embeddings": [
    { "nodeId": "node_abc", "embedding": [0.012, -0.034, 0.056, ...] },
    { "nodeId": "node_def", "embedding": [-0.023, 0.045, 0.078, ...] }
  ],
  "found": 2,
  "requested": 3
}
```

Nodes without cached embeddings are omitted from the response. Compare `found` vs `requested` to determine which nodes need embedding generation.

### Delete Map Embeddings

Remove all cached embeddings for a specific map.

```
DELETE /api/embeddings/:mapId
```

**Response:**
```json
{
  "success": true,
  "mapId": "clx1map001"
}
```

Uses Redis pattern matching (`emb:map:{mapId}:*`) to delete all node embeddings for the map in a single operation.
