# Authentication

NeuroCanvas uses [NextAuth.js](https://next-auth.js.org/) (integrated via `@sidebase/nuxt-auth`) for authentication. The system supports multiple providers and uses JWT-based sessions backed by a PostgreSQL database through the Prisma adapter.

## Auth Flow Overview

```
User clicks "Sign In"
    |
    v
Provider selection (Google, GitHub, Email, or Credentials)
    |
    v
OAuth redirect / Credentials validation / Magic link
    |
    v
NextAuth callback -> JWT created -> Session cookie set
    |
    v
User redirected to /dashboard (new users) or previous page
```

### JWT Session Strategy

NeuroCanvas uses the `jwt` session strategy rather than database sessions. This means:

- Sessions are stored as signed JWT tokens in HTTP-only cookies
- No server-side session lookup is needed on each request
- Sessions expire after **4 hours** (`maxAge: 14400`)
- The JWT includes `id`, `email`, `name`, and `picture` claims

### Database Adapter

The Prisma adapter persists users, accounts, and verification tokens to PostgreSQL. This enables:

- Linking multiple OAuth providers to a single user account
- Storing hashed passwords for credentials-based auth
- Email verification token management

## Available Providers

### Google OAuth

Sign in with Google using the OAuth 2.0 authorization code flow.

- Requires `GOOGLE_CLIENT_ID` and `GOOGLE_CLIENT_SECRET` environment variables
- Requests offline access with `access_type: "offline"` for refresh tokens
- Email account linking is enabled (`allowDangerousEmailAccountLinking: true`)

### GitHub OAuth

Sign in with GitHub using OAuth.

- Requires `GITHUB_CLIENT_ID` and `GITHUB_CLIENT_SECRET` environment variables
- Requests `read:user` and `user:email` scopes
- Automatically fetches primary email from the GitHub emails API if not included in the profile
- Email account linking is enabled

### Email (Magic Link)

Passwordless sign-in via email using [Resend](https://resend.com/) for transactional email delivery.

- Requires `RESEND_API_KEY` and email sender configuration
- Sends a branded HTML email with a one-click sign-in link
- Links expire after 24 hours
- Only available when Resend is configured

### Credentials (Email + Password)

Traditional email and password authentication.

- Always available regardless of environment configuration
- Passwords are hashed with bcrypt (12 salt rounds)
- Password requirements:
  - Minimum 8 characters
  - At least one uppercase letter
  - At least one lowercase letter
  - At least one number

## API Endpoints

### Get Current Session

Retrieve the currently authenticated user's session.

```
GET /api/auth/session
```

**Response (authenticated):**
```json
{
  "user": {
    "id": "clx1abc2d0001",
    "email": "user@example.com",
    "name": "Jane Doe",
    "image": "https://lh3.googleusercontent.com/..."
  },
  "expires": "2025-06-15T14:30:00.000Z"
}
```

**Response (unauthenticated):**
```json
{}
```

### List Available Providers

Returns the authentication providers configured on the server.

```
GET /api/auth/providers
```

**Response:**
```json
{
  "google": {
    "id": "google",
    "name": "Google",
    "type": "oauth",
    "signinUrl": "/api/auth/signin/google",
    "callbackUrl": "/api/auth/callback/google"
  },
  "github": {
    "id": "github",
    "name": "GitHub",
    "type": "oauth",
    "signinUrl": "/api/auth/signin/github",
    "callbackUrl": "/api/auth/callback/github"
  },
  "credentials": {
    "id": "credentials",
    "name": "Credentials",
    "type": "credentials",
    "signinUrl": "/api/auth/signin/credentials",
    "callbackUrl": "/api/auth/callback/credentials"
  }
}
```

The response only includes providers whose environment variables are configured. For example, Google will not appear if `GOOGLE_CLIENT_ID` is missing.

### Sign In

Initiate a sign-in flow. The method depends on the provider type.

#### OAuth Providers (Google, GitHub)

```
POST /api/auth/signin/:provider
```

Redirects the user to the OAuth provider's authorization page. After consent, the user is redirected back to `/api/auth/callback/:provider`, which completes the JWT exchange and sets the session cookie.

#### Credentials Provider

```
POST /api/auth/signin/credentials
Content-Type: application/json

{
  "email": "user@example.com",
  "password": "SecureP4ss"
}
```

**Success:** Sets session cookie and redirects to `/dashboard`.

**Failure (401):**
```json
{
  "error": "CredentialsSignin",
  "url": "/auth/error?error=CredentialsSignin"
}
```

### Register (Credentials)

Create a new account with email and password. This is a custom endpoint outside of the NextAuth.js flow.

```
POST /api/auth/register
Content-Type: application/json

{
  "email": "user@example.com",
  "password": "SecureP4ss",
  "name": "Jane Doe"
}
```

**Success (200):**
```json
{
  "success": true,
  "user": {
    "id": "clx1abc2d0001",
    "email": "user@example.com",
    "name": "Jane Doe",
    "createdAt": "2025-06-15T10:30:00.000Z"
  }
}
```

**Errors:**

| Status | Condition |
|---|---|
| `400` | Missing email or password |
| `400` | Invalid email format |
| `400` | Password does not meet requirements |
| `409` | Account with this email already exists |

### Sign Out

Terminate the current session.

```
POST /api/auth/signout
```

Clears the session cookie and redirects to `/auth/signin`.

### CSRF Token

Retrieve a CSRF token for use in form submissions.

```
GET /api/auth/csrf
```

**Response:**
```json
{
  "csrfToken": "abc123..."
}
```

## Session Usage in API Requests

All protected server routes validate the session using `getServerSession(event)` from the `#auth` module:

```typescript
import { getServerSession } from '#auth'

export default defineEventHandler(async (event) => {
  const session = await getServerSession(event)

  if (!session?.user?.email) {
    throw createError({
      statusCode: 401,
      statusMessage: 'Unauthorized'
    })
  }

  // Access session.user.id, session.user.email, etc.
})
```

The session cookie is automatically included in browser requests. No manual `Authorization` header is needed for web app usage.

## User Profile Endpoints

### Get Profile

```
GET /api/user/profile
```

Returns the authenticated user's profile along with their linked OAuth providers.

**Response:**
```json
{
  "id": "clx1abc2d0001",
  "email": "user@example.com",
  "name": "Jane Doe",
  "image": "https://lh3.googleusercontent.com/...",
  "emailVerified": "2025-06-01T00:00:00.000Z",
  "createdAt": "2025-06-01T00:00:00.000Z",
  "updatedAt": "2025-06-15T10:30:00.000Z",
  "providers": ["google", "github"]
}
```

### Update Profile

```
PATCH /api/user/profile
Content-Type: application/json

{
  "name": "Jane Smith",
  "image": "https://example.com/avatar.jpg"
}
```

Both fields are optional. Name is capped at 100 characters. Image must be a valid URL.

**Response:**
```json
{
  "id": "clx1abc2d0001",
  "email": "user@example.com",
  "name": "Jane Smith",
  "image": "https://example.com/avatar.jpg",
  "updatedAt": "2025-06-15T11:00:00.000Z"
}
```

### Delete Account

Permanently delete the authenticated user's account and all associated data (maps, sessions, OAuth accounts). This action cascades through all related Prisma models.

```
DELETE /api/user/account
Content-Type: application/json

{
  "confirmation": "DELETE MY ACCOUNT"
}
```

The exact string `"DELETE MY ACCOUNT"` must be provided as confirmation. Any other value returns a `400` error.

**Response:**
```json
{
  "success": true,
  "message": "Account deleted successfully"
}
```

## Security Notes

- All session cookies use `HttpOnly`, `SameSite=Lax`, and `Secure` (in production)
- PKCE code verifier and state cookies have a 15-minute TTL (`maxAge: 900`)
- The `AUTH_SECRET` environment variable is required and must be a strong random string
- OAuth tokens are stored encrypted in the database via the Prisma adapter
- Credentials passwords are hashed with bcrypt using 12 salt rounds
- The `trustHost: true` option is enabled for compatibility with reverse proxy deployments (Railway, Docker)
